## Purpose

This is the current run record for the Daytona sandbox forking developer guide. It replaces the older fork-counter dry run as the main narrative artifact.

The demo is more technical than a generic parallel-agent example. It shows that an AI agent can prepare meaningful state once, then reuse or branch that state with different lifecycle operations.

Main sentence:

> Stopping lets you return to the same sandbox. Pausing freezes live runtime state. Snapshotting saves a reusable baseline. Forking creates a new independent future from the current state.

## Post-Fix Validator Fan-Out

This run now incorporates the feedback that forking is more compelling when the parent has changed disk or memory state before the fork.

The demo keeps the original "one agent explores once, then workers specialize" story, but strengthens it:

1. The contract branch writes a candidate fix artifact.
2. That changed state becomes `contract-candidate-fix`.
3. Four workers fork from the changed state:
   - `api-design-review`
   - `implementation-review`
   - `regression-test-builder`
   - `checking-agent`
4. The checking agent aggregates validator reports.
5. If a worker does not green-light the fix, its report goes back to the original contract-fix branch.

The live runner also tries a memory-state probe. It warms a small process before forking `memory-state-check`, then logs whether the child can observe that warmed process. The guide should report the observed result conservatively.

## Lifecycle Semantics Covered

- Stop/start: same sandbox identity; filesystem state persists; running processes should not be treated as preserved.
- Pause: same sandbox identity; live process and memory state preservation where supported.
- Fork: new sandbox identity; same disk state at fork point; independent divergence.
- Snapshot: reusable prepared baseline.
- Restore/create from snapshot: new sandbox created later from saved baseline.
